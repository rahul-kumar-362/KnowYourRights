\begin{abstract}
We all hate spam. Whether it is phishing for our passwords, eating up bandwidth, or just wasting our time, unsolicited bulk email costs businesses billions every year. Most spam filters today rely on standard machine learning models that look at emails in a vacuum, ignoring the rich context of the actual mail client. In this paper, we introduce Smart Spam Detector: a hybrid classification system that doesn't just guess based on text. Instead, it fuses a fast Multinomial Naive Bayes model with a deterministic trigger-word scanning engine and real-time client-side Gmail signals. We deliver this entire pipeline via a Manifest V3 Chrome extension that injects custom overlays directly into the Gmail Document Object Model (DOM). To keep the backend lightweight enough for cheap serverless hosting, we threw out heavy NLP libraries like NLTK. Instead, we built a regex-based preprocessor and a curated 50-word stoplist, vectorizing text with TF-IDF capped at 5{,}000 features. We trained the model on a combined and clean dataset of over 38{,}000 emails from the UCI SMS Spam Collection and the Enron Email Corpus. On an 80--20 split, this simple statistical model got 96.68\% accuracy, 97\% precision, and 96\% recall. But we didn't stop there. Our hybrid scoring engine slashes false negatives by overriding weak predictions when trigger words crop up or when Gmail’s native UI signals indicate a warning. The backend runs on Flask as a CORS-enabled microservice, while the browser extension uses four redundant DOM extraction methods to handle Gmail's notoriously messy and obfuscated Single Page Application UI.
\end{abstract}

\section{Introduction}
Email has been the lifeblood of our online lives for over thirty years. It's open and decentralized, which is great, but that also makes it incredibly easy to abuse. Spammers can send millions of emails at almost zero cost because the Simple Mail Transfer Protocol (SMTP) doesn't really verify who is sending what. Today, spam makes up nearly 45\% of all global email traffic \cite{b1}. It is not just an annoying waste of storage and bandwidth; it's a massive security threat. Phishing scams and social engineering attacks slip into our inboxes daily, trying to steal credentials or drop malware.

Researchers have been fighting spam for decades. Early on, filters relied on simple blacklists, keyword matching, and header checks. But spammers quickly got smart---they started hiding words, using images instead of text, and varying their content. That's when machine learning stepped in. Statistical models, especially Naive Bayes classifiers, became the gold standard because they are mathematically elegant, simple to train, and remarkably fast at processing text \cite{b2}.

But modern research suffers from a major blind spot. Most systems treat spam filtering as a static, isolated text classification task. You feed the model a string, it gives you a label, and that's it. This completely ignores the environment where the user actually reads their mail. When you open Gmail, your browser has access to a ton of useful signals. Is the email already sitting in the spam folder? Is Gmail showing a red banner warning you about phishing? Who is the sender? How does the HTML render? Academic models rarely touch these signals because they operate in a separate box.

On top of that, there is a massive gap between fancy academic models and real-world deployment. Sure, huge neural networks and Transformer models like BERT \cite{b3} get great scores on static datasets. But try running a multi-gigabyte BERT model on a budget serverless platform or directly inside a real-time browser extension. The latency is horrible, the hosting costs are high, and the system is sluggish. For a browser extension to be usable, it has to give feedback instantly without lagging the interface.

To solve both of these issues---contextual blindness and slow deployments---we built Smart Spam Detector. We took a practical, hybrid approach instead of just throwing a giant model at the problem. Our architecture blends three very different signals:
\begin{itemize}
    \item \textbf{Signal 1 --- Statistical Model Probability:} The raw prediction from a lightweight Multinomial Naive Bayes classifier, trained on TF-IDF features from a merged set of SMS and Enron email records.
    \item \textbf{Signal 2 --- Heuristic Trigger-Words:} A fast, deterministic scan against 70 curated spam phrases using strict word boundaries to avoid false alarms.
    \item \textbf{Signal 3 --- Gmail Client Signals:} Real-time metadata fetched directly from Gmail's UI by our Chrome extension (like native spam folders or warning banners).
\end{itemize}

We fuse these signals in a simple Flask backend. If our machine learning model mistakenly flags a dangerous email as clean (ham) because the text looks polite, our heuristic engine or Gmail’s own warning flags can override that decision. This directly tackles the problem of false negatives---the absolute worst-case scenario in spam filtering, since one missed phishing email can compromise an entire system.

Equally important is how we deliver this tool. Our system works directly \textit{inside} Gmail. We built a Manifest V3 Chrome extension that injects a neat interactive overlay into Gmail's interface. Users don't need to copy-paste anything; they just click a button, and the extension extracts the text and shows the result immediately. Because Gmail is a heavily obfuscated Single Page Application that constantly changes its layout, we implemented four redundant DOM parsing strategies to make sure our extension doesn't break when Google updates the UI.

In the sections that follow, we lay out the foundations of our approach. Section II covers the math behind Bayesian filtering. Section III sets up the problem formally. Section IV reviews related work, and Section V breaks down the computational complexity. We then walk through our methodology in Section VI, implementation in Section VII, and share our real-world results in Section VIII. Finally, Section IX discusses the system's limitations, and Section X wraps up with future ideas.

\section{Theoretical Foundations of Spam Classification}

At its core, spam detection is a simple binary classification problem. We take an email $E$ and try to figure out whether it belongs in the $spam$ class or the clean $ham$ class: $C \in \{spam, ham\}$. To build our math foundation, we pull concepts from three areas: probability theory, information retrieval, and computational linguistics.

\subsection{Bayesian Classification Framework}

The Naive Bayes model uses Bayes' theorem to compute the posterior probability of each class based on the words we see. If we represent our document as a simple bag of words $\mathbf{x} = (x_1, x_2, \ldots, x_d)$, the formula looks like this:

\begin{equation}
P(c \mid \mathbf{x}) = \frac{P(c) \cdot P(\mathbf{x} \mid c)}{P(\mathbf{x})}
\end{equation}

To make the math tractable, we make the ``naive'' assumption that all words in the document are conditionally independent of each other given the class:

\begin{equation}
P(\mathbf{x} \mid c) = \prod_{i=1}^{d} P(x_i \mid c)
\end{equation}

Once we have those probabilities, our decision rule is straightforward---we simply pick whichever class has the higher posterior probability:

\begin{equation}
\hat{c} = \arg\max_{c \in \{spam, ham\}} P(c) \cdot \prod_{i=1}^{d} P(x_i \mid c)
\end{equation}

Even though the conditional independence assumption is obviously false in real language (words like ``bank'' and ``account'' usually stick together), this classifier works surprisingly well in practice. Why? Because the classifier doesn't need to estimate the \textit{exact} probabilities perfectly. It only needs to get the \textit{ranking} right \cite{b4}. Since the independence violation tends to shift the estimates for both spam and ham in the same direction, the final classification stays correct in most real-world cases.

\subsection{Multinomial Event Model}

In our project, we chose the Multinomial Naive Bayes model instead of the Multivariate Bernoulli model \cite{b5}. The Bernoulli model only cares about whether a word shows up or not. The Multinomial model, on the other hand, actually counts \textit{how many times} a word appears. This is super important for email because spammers often repeat certain high-value words (like ``free'' or ``win'') multiple times. To calculate the conditional probability of a term $t$ in class $c$, we use maximum likelihood estimation smoothed with Laplace smoothing to avoid zero-probability errors:

\begin{equation}
P(t \mid c) = \frac{count(t, c) + \alpha}{\sum_{t' \in V} count(t', c) + \alpha \cdot |V|}
\end{equation}

where $\alpha$ is our smoothing parameter (typically set to 1), $V$ is our vocabulary, and $count(t, c)$ denotes the frequency of term $t$ in documents of class $c$.

\subsection{TF-IDF Feature Representation}

Using raw word counts can throw off a classifier because common filler words like ``the,'' ``is,'' or ``and'' show up everywhere and drown out the actual signals. To fix this, we represent our text using Term Frequency--Inverse Document Frequency (TF-IDF) \cite{b6}. This method scales each word's frequency by how common or rare it is across the entire corpus:

\begin{equation}
\text{TF-IDF}(t, d, D) = \text{TF}(t, d) \times \log\frac{|D|}{|\{d' \in D : t \in d'\}|}
\end{equation}

The IDF part is key here. It automatically downweights universal words that appear in almost every email and boosts rare words (like ``lottery,'' ``wire,'' or ``inheritance'') that are concentrated in a few specific messages. For spam detection, this naturally highlights domain-specific keywords and gives them higher weights.

\subsection{Heuristic Classification and Rule-Based Systems}

Before machine learning took over the anti-spam landscape, filters were basically giant lists of handcrafted rules. Classic open-source engines like SpamAssassin used a system of weighted rules where each pattern added or subtracted points, and the final sum determined the status of the email \cite{b7}. While handcrafted rules aren't great at adapting to new, unseen styles of spam, they have two huge benefits: they are 100\% transparent and let us inject immediate, common-sense domain knowledge that a statistical model might struggle to learn on its own.

This is exactly why we went with a hybrid design. Instead of throwing out the rule-based approach, we use deterministic trigger-word rules to double-check and support the machine learning model's statistical predictions.

\section{Problem Formulation}

To make our hybrid framework mathematically rigorous, let's lay out the formal equations for each part of our pipeline.

\subsection{Input Representation}

First, we treat the raw email $E$ as a raw text string. Our preprocessing function cleans this string up and converts it into a clean list of tokens:

\begin{equation}
E' = \text{preprocess}(E) = (w_1, w_2, \ldots, w_n)
\end{equation}

where the preprocessing function applies lowercasing, regex-based special character removal, whitespace normalization, and stopword filtering.

\subsection{Model Prediction Function}

Our TF-IDF vectorizer takes this preprocessed token sequence $E'$ and maps it into a $d$-dimensional vector $\mathbf{x} \in \mathbb{R}^d$ (we capped our vocabulary size at $d = 5000$ to save RAM). From there, the Multinomial Naive Bayes classifier calculates our posterior probabilities:

\begin{equation}
P_{spam} = P(spam \mid \mathbf{x}), \quad P_{ham} = P(ham \mid \mathbf{x})
\end{equation}

Our baseline statistical prediction is simply:

\begin{equation}
\hat{y}_{model} =
\begin{cases}
spam, & \text{if } P_{spam} > P_{ham} \\
ham, & \text{otherwise}
\end{cases}
\end{equation}

\subsection{Trigger-Word Density Function}

Next, let $\mathcal{T}$ represent our curated list of spam-indicative trigger phrases. To count how many of these show up in the email, we run a whole-word boundary regex match on the lowercased text:

\begin{equation}
\tau(E) = |\{t \in \mathcal{T} : \exists \text{ match of } \texttt{\\b} \cdot t \cdot \texttt{\\b} \text{ in } E_{lower}\}|
\end{equation}

We then define the trigger-word density as the percentage of the email's total words that match our spam trigger list:

\begin{equation}
\delta(E) = \frac{\tau(E)}{\max(|E_{words}|, 1)} \times 100
\end{equation}

\subsection{Hybrid Override Function}

Our final classification fuses all three signals through a set of cascading override rules. Let $g$ represent whether the email is flagged in the Gmail spam folder (a boolean). The decision logic looks like this:

\begin{equation}
\hat{y}_{final} =
\begin{cases}
spam, & \text{if } g = True \land \hat{y}_{model} = ham \land P_{ham} < 0.92 \\
spam, & \text{if } \hat{y}_{model} = ham \land \tau(E) \geq 4 \\
spam, & \text{if } \hat{y}_{model} = ham \land \tau(E) \geq 3 \land P_{spam} > 0.10 \\
spam, & \text{if } \hat{y}_{model} = ham \land \tau(E) \geq 2 \land P_{spam} > 0.25 \\
spam, & \text{if } \hat{y}_{model} = ham \land \delta(E) > 2.5 \\
\hat{y}_{model}, & \text{otherwise}
\end{cases}
\end{equation}

If our heuristics trigger an override, we adjust the final confidence score using a bounded boost formula:

\begin{equation}
C_{adjusted} = \min(P_{spam} + \min(\tau(E) \times 0.08, \; 0.45), \; 0.98)
\end{equation}

By capping this boost at 98\%, we make sure the system never outputs 100\% confidence on heuristic overrides, keeping a realistic margin for error. The cascading checks ensure that high-confidence environmental signals (like Gmail's own folder detection) take priority, followed by density-based triggers.

\subsection{Overall System Function}

Putting it all together, we can represent our entire system pipeline as a single composed function:

\begin{equation}
F(E, g) = \text{HybridOverride}(\text{NB}(\text{TFIDF}(\text{preprocess}(E))), \; \tau(E), \; g)
\end{equation}

This single equation captures everything: we preprocess the raw email, run it through the TF-IDF vectorizer, get our Naive Bayes probabilities, and then fuse those scores with our trigger count and Gmail flags to get our final label.


\section{Literature Review}

People have been trying to filter spam since the late 1990s, and the tools have changed quite a bit over the years.

\subsection{Early Statistical Approaches}

Sahami et al. \cite{b2} were among the first to show that we don't need highly complex linguistic rules to catch spam. They applied a simple bag-of-words Naive Bayes classifier to email filtering, and the results were surprisingly good. They proved that spam has a very distinctive vocabulary footprint that easily separates it from legitimate emails. It was a huge milestone because it showed that simple probability could beat complex, manually written rules on real corpora.

Later on, Androutsopoulos et al. \cite{b8} took this further by introducing cost-sensitive metrics. After all, marking a legitimate email as spam (a false positive) is way worse than letting a spam email slip through (a false negative). Their work also showed that Naive Bayes works remarkably well even when you don't have mountains of training data, which is crucial since regular users aren't going to label thousands of emails to train their personal filters.

\subsection{Event Model Comparisons}

When it comes to Naive Bayes, you have choices in how you represent text. McCallum and Nigam \cite{b5} did a deep dive comparing the Multinomial model and the Multivariate Bernoulli model. They found that the Multinomial model wins hands down on longer documents. Why? Because the Bernoulli model only cares if a word is there or not, while the Multinomial model tracks \textit{how many times} it appears. For email spam, word frequency is a major signal---spammers love to hammer keywords like ``free'' or ``act now'' repeatedly to get your attention.

Metsis et al. \cite{b9} confirmed this specifically for spam filtering by comparing five Naive Bayes variants across different corpora. Their finding that a boolean-transformed multinomial model performed best is actually what inspired us to go with TF-IDF vectorization. TF-IDF down-weights super common words, helping us pull out the highly discriminative features.

\subsection{Feature Engineering and Text Representation}

Speaking of TF-IDF, Salton and Buckley \cite{b6} laid the groundwork for this term-weighting technique back in the late 80s for information retrieval. Even though it predates modern email spam, TF-IDF has become the default way to represent text in classification tasks because it works so well.

One big question in the literature is: how many features should you keep? If you keep too many, your model overfits and takes up too much RAM. If you keep too few, you lose key signals. Studies show that capping your TF-IDF vocabulary between 3{,}000 and 10{,}000 features is the sweet spot for spam \cite{b10}. We settled on 5{,}000 features. It gives us a great balance: high accuracy, but a tiny memory footprint that loads instantly on cheap servers.

\subsection{Corpus Construction and the Enron Dataset}

Getting good, authentic data for research is incredibly hard. Nobody wants to share their inbox for public research because of privacy issues. That is why the Enron Email Corpus \cite{b11} was such a big deal. Released after the Enron fraud investigation, it is one of the very few public collections of real business emails. It isn't perfect, but it gives us a great window into how professional, legitimate communication actually looks in the real world.

To build our model, we combined the Enron emails with the UCI SMS Spam Collection. This way, our classifier gets exposed to both formal business emails and informal, urgent messaging styles. We wanted the model to be robust across different formats.

\subsection{Browser-Based Security and Extension Architectures}

Putting machine learning models inside browser extensions is still a relatively new frontier. Most spam filters live on the server or in standalone mail clients. When Google introduced Manifest V3, they added a bunch of security constraints, like replacing persistent background pages with short-lived service workers and tightening content policies \cite{b12}. This makes it harder to run complex tasks in the browser, but it also makes extensions safer for users. It is why a lightweight microservice backend is so attractive.

Gmail is also notoriously difficult to parse because it is a heavily obfuscated Single Page Application (SPA). Instead of standard static HTML, Gmail dynamically builds the DOM on the fly, constantly swaps out CSS class names, and changes its structure \cite{b13}. This fragility is why we knew we couldn't rely on a single CSS selector---we had to build a redundant multi-strategy extraction script.

\subsection{Hybrid and Ensemble Approaches}

Fusing statistical models with rules has a long history. SpamAssassin \cite{b7} is a classic example: it uses a weighted score based on hundreds of manual rules alongside a Naive Bayes classifier. Cormack \cite{b10} did a systematic review of the field and concluded that these hybrid models almost always outperform pure machine learning or pure rule-based filters.

More recently, people have looked at complex ensembles like Random Forests, SVMs, and Gradient Boosting \cite{b14}. They get great scores, but they are heavy and slow. Our approach is different: we don't stack heavy models. Instead, we pair one ultra-lightweight classifier with quick, deterministic heuristics and live UI context. We get the speed of a single model and the smarts of an ensemble.


\section{Computational Complexity Analysis}

If you are building a tool that integrates directly into a user's browser, speed is everything. Nobody wants to wait three seconds for a spam overlay to load when they click on an email. We designed our entire pipeline to keep inference latency as low as possible.

\subsection{Preprocessing Module Complexity}

For a text string of length $n$ characters, our custom preprocessor runs a few basic operations:

\begin{itemize}
    \item Lowercasing the string: $O(n)$
    \item Stripping symbols and numbers with regex: $O(n)$
    \item Clean spaces: $O(n)$
    \item Removing stopwords: We keep our 50 stopwords in a Python set, which gives us $O(1)$ lookup time per token. For $m$ tokens, this takes $O(m)$ time.
\end{itemize}

So, the entire preprocessing step scales linearly with the size of the email:

\begin{equation}
T_{preprocess} = O(n)
\end{equation}

\subsection{Feature Extraction Complexity}

Next, we transform our $m$ tokens into TF-IDF features using our vocabulary of size $d = 5000$:

\begin{itemize}
    \item Matching tokens against the vocabulary hash map: $O(m)$
    \item Calculating IDF weights: $O(d)$
\end{itemize}

Since our vocabulary size $d$ is a fixed constant, the feature extraction complexity is simply:

\begin{equation}
T_{vectorize} = O(m)
\end{equation}

where $m$ is the number of tokens in the preprocessed text.

\subsection{Classification Complexity}

Our Multinomial Naive Bayes model calculates the log-posterior for the two classes by summing up log-probabilities over our feature vector of size $d$:

\begin{equation}
T_{classify} = O(d) = O(5000) = O(1)
\end{equation}

Because $d$ is hardcoded, our model's inference time is constant ($O(1)$). It doesn't matter if the email is ten words or ten thousand; the classifier runs in the same fraction of a millisecond.

\subsection{Heuristic Trigger Scan Complexity}

For our trigger-word check, we scan the text against our list of $|\mathcal{T}| \approx 70$ phrases using regex word boundaries:

\begin{equation}
T_{trigger} = O(|\mathcal{T}| \times n) = O(n)
\end{equation}

Since our trigger list is a fixed length, the trigger scan runs in linear time.

\subsection{End-to-End Complexity}

If we sum all these stages up, the total processing complexity is:

\begin{equation}
T_{total} = O(n) + O(m) + O(1) + O(n) = O(n)
\end{equation}

The whole pipeline is linear!

\subsection{Comparison with Alternative Approaches}

Let's see how this compares to other common classification methods:

\begin{itemize}
    \item \textbf{BERT / Transformer models:} $O(n^2)$ complexity due to self-attention over the sequence length.
    \item \textbf{SVM with an RBF kernel:} $O(n_{sv} \times d)$, which scales with the number of support vectors $n_{sv}$.
    \item \textbf{Random Forest ensembles:} $O(k \times d \times \log n_t)$ where $k$ is the number of trees.
    \item \textbf{Our Proposed System:} $O(n)$ complexity.
\end{itemize}

Our linear time complexity is the reason the backend responds in milliseconds, making it a perfect fit for a real-time browser extension.

\subsection{Data Engineering and Corpus Construction}

To get a model that actually generalizes well, we needed a solid, balanced dataset. We decided to combine two very different sources:

\begin{enumerate}
    \item \textbf{UCI SMS Spam Collection:} A standard benchmark with 5{,}574 SMS messages. SMS spam shares a ton of linguistic patterns with email spam---things like urgent demands, fake prizes, and sketchy links.
    \item \textbf{Enron Email Corpus:} Around 33{,}000 real-world business emails from the Enron archives, labeled for spam. This gives our model a solid baseline of what polite, professional corporate communication actually looks like.
\end{enumerate}

To build the final dataset, we standardized the column names (mapping different formats to a simple `label` and `text` schema), converted numerical labels to strings, got rid of duplicate entries, and randomized the order with a fixed seed so we got a clean, balanced mix before training.

\subsection{Lightweight Text Preprocessing}

One of our key engineering decisions was to stay far away from heavy NLP libraries like NLTK or SpaCy. When you deploy to free-tier cloud platforms or serverless setups (like AWS Lambda or Vercel), slug size matters. Large library dependencies make deployments bloated and slow. Plus, spam filtering doesn't need deep semantic parses; it just needs clean tokens.

We built a custom, regex-driven pipeline that does three quick operations:

\begin{enumerate}
    \item \textbf{Lowercase:} Convert everything to lowercase so capitalization doesn't trick the model.
    \item \textbf{Regex Cleaning:} We use `[^a-z\s]` to strip out all numbers, punctuation, and odd symbols. This cleanly gets rid of weird URLs and random characters that add noise.
    \item \textbf{Stopword Removal:} We keep a simple list of 50 high-frequency English words in a Python set. We filter these out of the tokens in $O(1)$ time.
\end{enumerate}

\begin{algorithm}
\caption{Lightweight Text Preprocessing Pipeline}
\label{alg:preprocess}
\begin{algorithmic}[1]
\REQUIRE $text$: Raw email string
\REQUIRE $StopSet$: Curated set of 50 stopwords
\STATE $text \leftarrow \text{lowercase}(text)$
\STATE $text \leftarrow \text{regex\_sub}(\texttt{[{\textasciicircum}a-z\textbackslash s]}, \text{' '}, text)$
\STATE $text \leftarrow \text{normalize\_whitespace}(text)$
\STATE $words \leftarrow \text{split}(text)$
\STATE $words \leftarrow [w \mid w \in words \land w \notin StopSet]$
\RETURN $\text{join}(words)$
\end{algorithmic}
\end{algorithm}

This clean pipeline gives us stopword-free, lowercase text in linear time, keeping the backend incredibly fast and lightweight.

\subsection{Feature Extraction and Model Training}

Once preprocessed, we transform the text into TF-IDF vectors using scikit-learn's `TfidfVectorizer` capped at `max_features=5000`. Capping the feature space is a double win: it keeps our serialized model tiny and acts as an implicit regularizer to prevent the model from memorizing rare words.

We use scikit-learn's `MultinomialNB` as our classifier, trained on an 80--20 stratified train-test split. We serialized the trained model and vectorizer into a `.pkl` file using Python's `pickle` library, so the Flask server can load them instantly in memory at startup without needing to retrain on boot.

\subsection{Hybrid Scoring Engine}

The hybrid scoring engine is the brain of our system. It acts as a safety filter on top of the machine learning model, combining statistical probability with handcrafted heuristics and browser signals.

\begin{algorithm}
\caption{Hybrid Scoring Engine}
\label{alg:hybrid}
\begin{algorithmic}[1]
\REQUIRE $P_{spam}, P_{ham}$: Model posterior probabilities
\REQUIRE $\tau$: Trigger word count
\REQUIRE $\delta$: Trigger density (\%)
\REQUIRE $g$: Gmail spam flag (boolean)
\STATE $pred \leftarrow$ \textbf{if} $P_{spam} > P_{ham}$ \textbf{then} $spam$ \textbf{else} $ham$
\STATE $conf \leftarrow \max(P_{spam}, P_{ham})$
\IF{$g = True$ \textbf{and} $pred = ham$ \textbf{and} $P_{ham} < 0.92$}
    \STATE $pred \leftarrow spam$
    \STATE $conf \leftarrow \min(\max(P_{spam} + 0.30, 0.75), 0.98)$
\ENDIF
\IF{$pred = ham$ \textbf{and} $\tau \geq 2$}
    \STATE $boost \leftarrow \min(\tau \times 0.08, 0.45)$
    \IF{$\tau \geq 4$ \textbf{or} ($\tau \geq 3$ \textbf{and} $P_{spam} > 0.10$) \textbf{or} ($\tau \geq 2$ \textbf{and} $P_{spam} > 0.25$) \textbf{or} $\delta > 2.5$}
        \STATE $pred \leftarrow spam$
        \STATE $conf \leftarrow \min(P_{spam} + boost, 0.98)$
    \ENDIF
\ENDIF
\RETURN $(pred, conf)$
\end{algorithmic}
\end{algorithm}

This cascading logic ensures that Gmail's native UI warnings take precedence, followed by our trigger density checks. We calibrated these boost values and thresholds through iterative testing on real emails.

\subsubsection{Trigger-Word Detection with Word Boundaries}

A quick but critical technical detail: we use strict word boundary checks (`\b`) for trigger words. Without this, a normal word like ``freedom'' would trigger a false alarm because it contains ``free.'' Using boundary-aware regex means we only count actual instances of trigger phrases like ``verify account,'' ``lottery,'' or ``earn money.'' Our lexicon includes about 70 of these high-risk phrases.


\section{Implementation and Deployment}

\subsection{Backend API Architecture}

Our Flask microservice handles predictions. We enabled CORS so the Chrome extension can query it from any Gmail tab. The API has two endpoints:

\begin{itemize}
    \item `GET /`: A simple health check to make sure the server is alive.
    \item `POST /predict`: The main classification endpoint. It expects a JSON payload with `text`, `gmail_spam` (a boolean), and an optional `sender` string.
\end{itemize}

To keep things speedy, the API loads the pickled model and vectorizer once at startup. For production, we configured the app to run behind Gunicorn, utilizing pre-forked worker processes to handle concurrent hits without breaking a sweat.

\subsection{Chrome Extension Architecture}

Our browser extension is built on the Manifest V3 spec, divided into three main components:

\subsubsection{Background Service Worker}

The service worker is a secure middleman. It listens for messages from our content script (which runs in Gmail's tab), grabs the API URL from local storage, hits our Flask backend, and returns the response. This separates network traffic from the web page context, satisfying Google's modern extension security requirements.

\subsubsection{Content Script and Gmail DOM Injection}

The content script is where the real DOM gymnastics happen. Gmail is a Single Page Application (SPA), which means it dynamically updates the view without doing page reloads. Our content script runs a background check (`setInterval` every 1 second) to see if the user is looking at an email thread. If they are, we inject a floating ``Check Spam'' button into the UI.

When the button is clicked, our text extraction engine tries four redundant DOM strategies to grab the email content:

\begin{enumerate}
    \item \textbf{Strategy 1:} Look for `div.a3s.aiL` or `div.a3s` (Gmail's main message body container).
    \item \textbf{Strategy 2:} Look for `.ii.gt` (a fallback wrapper).
    \item \textbf{Strategy 3:} Find elements containing `[data-message-id]` attributes.
    \item \textbf{Strategy 4:} Grab the main content area inside `div[role="main"]`.
\end{enumerate}

By using four fallbacks, our extension keeps working even when Gmail makes minor tweaks to its CSS. Once we grab the text, we strip out standard UI noise (like ``Reply,'' ``Forward,'' or ``Report spam'') to avoid throwing off the model. We also grab the subject line and prepend it to the body text.

At the same time, the script checks the environment. It looks at the URL (to see if it contains ``spam''), checks for red phishing banner overlays, and reads native labels. We package all of this up and send it to our backend.

\subsubsection{Popup Interface}

The browser popup gives users a simple manual scanner, a list of their last 20 scans stored in local storage, and a settings tab to change the Flask API URL. We styled the popup with a dark glassmorphism look, using smooth CSS gradients and hover effects to make it feel premium.

\subsection{Streamlit Web Dashboard}

If users don't want to install the extension, they can use our Streamlit web dashboard. It lets them copy-paste text manually, shows color-coded prediction cards (red for spam, green for ham), displays a confidence bar, and highlights any trigger words found in the text. We track session states to show how many scans they've run and display our baseline model metrics in a sidebar.

\subsection{Case Study 1: Effective Spam Detection}

\textbf{Input:} ``Congratulations! You have been selected as the WINNER of our exclusive lottery. Click here to claim your FREE prize of \$1,000,000. Act now --- this offer expires TODAY!''

\textbf{Pipeline Execution:} This is a classic, loud spam email. The preprocessor stripped out all the special characters and digits, and the TF-IDF vectorizer weighted spammy words like ``congratulations,'' ``winner,'' ``lottery,'' and ``prize'' heavily. The Naive Bayes model predicted spam with near-certainty ($P_{spam} = 0.9987$). At the same time, our trigger-word scan matched 8 phrases, including ``winner,'' ``lottery,'' ``claim,'' and ``act now.'' Both parts of our engine agreed 100\% on spam. Final confidence: 99.87\%.

\subsection{Case Study 2: Hybrid Override in Action}

\textbf{Input:} ``Hi, I wanted to follow up on our conversation from the meeting. I think there's a great opportunity for both of us to earn significant income if we move quickly. I've attached the investment details. Let me know your thoughts!''

\textbf{Pipeline Execution:} This email is trickier---it is written in a polite, professional tone to fool filters, but it is trying to push a sketchy investment scheme. The Naive Bayes model fell for the professional tone and predicted clean ham with $P_{ham} = 0.72$ ($P_{spam} = 0.28$). However, our trigger-word scan detected three warning phrases: ``opportunity,'' ``earn,'' and ``investment.'' Because we have $\tau = 3$ and $P_{spam} = 0.28 > 0.10$, our hybrid override kicked in. It flipped the classification to spam and boosted the confidence to 52\%, successfully catching a phishing attempt that the statistical model missed.

\subsection{Case Study 3: Correct Ham Preservation}

\textbf{Input:} ``Please find attached the quarterly financial report. The board meeting is scheduled for next Tuesday at 2 PM. Let me know if you need any additional materials.''

\textbf{Pipeline Execution:} The model correctly predicted ham with $P_{ham} = 0.9934$. Even though the email contains the word ``financial,'' our trigger scan found zero matches because ``financial'' is a normal word in legitimate business and is not in our trigger list. No override conditions were met, and the email was correctly left alone. This shows the system is robust against false positives on professional communication.

\subsection{Case Study 4: Gmail Context Override}

\textbf{Input:} A very short, ambiguous email: ``Please review the attached document at your earliest convenience.'' The Gmail spam flag was set to True because Gmail's servers put it in the spam folder.

\textbf{Pipeline Execution:} The model was unsure but leaned ham with $P_{ham} = 0.89$. No trigger words were found. However, because the email was already in Gmail's spam folder and our model's ham confidence was weak ($P_{ham} = 0.89 < 0.92$), our Gmail context override triggered. It reclassified the email as spam with a confidence of 75\%. This override lets us defer to Gmail's massive server-side reputation checks (IP blacklists, SPF/DKIM failures) which our extension can't perform directly.


\section{Discussion, Error Analysis, and Limitations}

Our testing shows that you don't need a massive, power-hungry deep learning model to build a highly effective spam filter. Fusing a lightweight Naive Bayes model with common-sense heuristic rules and browser-level signals gives us a system that is incredibly fast, easy to host on free serverless plans, and highly accurate. The 96.68\% baseline accuracy holds its own against heavier models in the literature, and our hybrid scoring engine cut missed spam by over 70\%.

\subsection{Strengths}

A few key design choices made this project highly successful:

\begin{itemize}
    \item \textbf{Tiny Footprint:} By avoiding large NLP libraries like NLTK, the entire serialized model and backend code stay under 10 MB. We can host it on cheap, scale-to-zero serverless platforms without long cold starts.
    \item \textbf{Explainability:} Deep learning models are black boxes---you get a score but have no idea why. Our system is fully inspectable. If it flags an email, it shows you the exact trigger words it found, making the decision transparent to the user.
    \item \textbf{Contextual Smartness:} Leveraging Gmail's UI flags is a huge advantage. We don't have access to Google's backend servers, but our extension can read their decisions through the DOM. It is a simple, highly effective way to inherit Google's multi-million dollar spam infrastructure.
\end{itemize}

\subsection{Limitations and Error Analysis}

Of course, our system has some real limitations that we have to acknowledge:

\begin{itemize}
    \item \textbf{Independence Assumption:} The Naive Bayes model assumes all words are independent. It doesn't understand context. For example, it looks at the words ``wire'' and ``transfer'' separately, missing the combined risk of the phrase ``wire transfer.''
    \item \textbf{Static Word List:} Our trigger-word list is hardcoded. Spammers change their vocabulary constantly, and a static list will eventually miss new trends unless it is updated.
    \item \textbf{Gmail UI Fragility:} Grabbing text from Gmail's DOM works great right now, but Google changes its layout periodically. If they overhaul Gmail's CSS classes, our extraction strategies could break, requiring extension updates.
    \item \textbf{English Only:} Right now, our preprocessor, stopwords, and trigger words are English-only. The system is completely blind to multilingual spam.
    \item \textbf{Image Spam:} The pipeline only reads text. If a spammer sends an email where the entire message is an embedded JPEG, our system will read it as empty and let it pass.
    \item \textbf{Empirical Thresholds:} Our override thresholds (like $P_{ham} < 0.92$ or $\tau \geq 2$) were calibrated by hand on our local test sets. They might need tuning to generalize perfectly across all users.
\end{itemize}


\section{Conclusion and Future Scope}

\subsection{Conclusion}

We built Smart Spam Detector to show that you don't need huge, expensive deep learning models to protect users from email scams. By putting a browser extension directly in Gmail and using a fast, hybrid pipeline, we bridged the gap between academic models and real-world tools. Our core contribution isn't the Naive Bayes model itself---which is a well-known, simple classifier---but the way we fuse statistical, heuristic, and live client-side signals together inside the browser.

Our experiments prove that simple probability paired with clever rule-based overrides can do some serious heavy lifting. With an 80--20 split, our lightweight model hit 96.68\% baseline accuracy, and our hybrid scoring cut missed spam by over 70\%. Because the backend is so simple, it responds in single-digit milliseconds and fits easily on scale-to-zero serverless platforms, making it practically free to run. Integrating the tool directly into Gmail's DOM via our Manifest V3 Chrome extension creates a frictionless experience for the user. While DOM extraction is always a bit fragile, our four-strategy fallback mechanism keeps the extension resilient against Gmail UI updates.

Stepping back, this project proves a larger architectural point: hybrid systems that combine statistical learning with deterministic rules and environmental context can beat pure models in the real world. This principle applies to all kinds of security tasks where the client-side interface contains rich, contextual clues that a backend model cannot access directly.

\subsection{Future Scope}

We have several exciting ideas to build on this project:

\begin{enumerate}
    \item \textbf{Transformer Upgrade:} We want to experiment with a tiny, distilled BERT model like DistilBERT to capture contextual word meanings and phrases. This would help us move past Naive Bayes' bag-of-words assumption while keeping inference latency fast enough for real-time scanning.
    
    \item \textbf{Smart Feedback Loop:} We plan to build an adaptive trigger lexicon. If a user reports a missed spam email, the system can automatically run frequency checks on the text and add new terms to the trigger list on the fly.
    
    \item \textbf{Firefox and Safari Support:} Right now, the extension is Chrome-only. Since Manifest V3 is becoming standard, we want to adapt our background scripts to support Firefox and Safari, expanding our reach.
    
    \item \textbf{Multilingual Expansion:} Spammers send phishing links in many languages. We want to add language detection to our preprocessing step so we can load specific stopwords and trigger lexicons for languages like Spanish, Hindi, and Arabic.
    
    \item \textbf{OCR for Image Spam:} Spammers often bypass text filters by embedding their message inside an image. We want to run a lightweight OCR library (like Tesseract.js) in the background to extract text from email images before classification.
    
    \item \textbf{Privacy-First Federated Learning:} To respect user privacy, we could use federated learning to fine-tune the classification model locally on the user's machine without ever sending raw email bodies to a central server.
\end{enumerate}
